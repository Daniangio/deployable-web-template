# Astralia Chronicles — Current Architecture (April 25, 2026)

This document is the **current implementation baseline** (what runs today), including the new distributed runtime path, storage model, and known limitations.

---

## 1) Runtime Topology

## 1.1 Services
- **Frontend**: React + Vite (`frontend/`)
- **Backend Gateway**: FastAPI + WebSocket endpoint (`backend/app/main.py`)
- **Backend Worker**: distributed match worker (`backend/app/worker_main.py`)
- **PostgreSQL**: persistent relational state
- **Redis**: command streams, pub/sub backplane, ephemeral runtime keys
- **Firebase Auth**: identity provider (client login + backend verification)

## 1.2 Local Compose default
- `docker-compose.yml` starts:
  - `backend` (gateway)
  - `backend-worker`
  - `postgres`
  - `redis`
  - `frontend`
- Optional ops tooling profile:
  - `redisinsight` (`--profile ops`)

## 1.3 Production Compose
- `deploy/docker-compose.yml` mirrors gateway/worker split.
- Optional `redisinsight` profile is available and bound to `127.0.0.1` (SSH tunnel workflow).

## 2) Backend Architecture

## 2.1 Gateway responsibilities
- Authenticate socket/API requests.
- Accept user actions.
- When distributed runtime is enabled, enqueue gameplay commands to Redis Streams.
- Forward outbound messages to connected users.
- Relay cross-node outbound messages via Redis pub/sub backplane.
- Lobby websocket payloads include room/user/invite lifecycle only (global lobby chat was removed).

Main code paths:
- `backend/app/websocket_gateway.py`
- `backend/app/main.py`
- `backend/app/connection_manager.py`
- `backend/app/distributed/gateway_backplane.py`

## 2.2 Worker responsibilities
- Consume command streams from Redis.
- Hydrate game runtime state when needed.
- Execute existing RoomManager/InGameService game logic.
- Emit responses/state updates through the same connection manager + backplane.
- Reclaim pending stream entries after worker restarts (`XAUTOCLAIM`) to reduce stuck commands.
- Apply retry/dead-letter policy for malformed or failing stream entries.
- Evict per-game runtime execution caches after each processed command (state rehydrates from Redis/DB on next command).

Main code paths:
- `backend/app/worker_main.py`
- `backend/app/distributed/local_runtime_bridge.py`

## 2.3 Transitional distributed layer
- Command envelope/contracts:
  - `backend/app/distributed/command_models.py`
  - `backend/app/distributed/contracts.py`
- Redis adapters:
  - `backend/app/distributed/redis_runtime.py`
  - CAS Lua script: `backend/app/distributed/scripts/cas_commit.lua`
  - timer actor: `backend/app/distributed/timer_actor.py`
  - replay archiver worker: `backend/app/distributed/replay_archiver.py`

Current command routing coverage:
- Match actions (`play_card`, `cast_action`, `attack`, `ability`, `retire`, `resolve_trigger`, etc.)
- Query-type realtime calls (`validate_game_action`, `request_action_targets`, `request_player_decision_context`, `request_game_state`, `request_distributed_debug`)

## 2.4 Command reliability path (current)
- Primary consumption path:
  - `XREADGROUP` on `game:stream:<game_id>`
- Recovery path:
  - `XAUTOCLAIM` reclaim of pending entries older than `REDIS_PENDING_CLAIM_IDLE_MS`
- Failure handling:
  - Retry in same stream with incremented `retry_count` until `REDIS_COMMAND_MAX_RETRIES`
  - Move to dead-letter stream `game:stream:dead_letter` when retry limit is exceeded
  - User-invalid actions (`TypeError`/`ValueError`, e.g. invalid undo timing) are returned as immediate websocket `error` and acknowledged (no retry loop)
- Commit discipline:
  - runtime snapshot CAS commit in Redis (`version`, `recent_command_ids`) via Lua script
- Note:
  - if requeue/dead-letter write itself fails, entry remains pending and is retried by reclaim path.

## 2.5 Client synchronization and diagnostics
- `game_state_update` payloads now include a monotonic `state_version` generated by backend runtime versioning.
- Frontend applies snapshot ordering using `state_version` first, with `replay_frame_count` as secondary tie-breaker.
- This prevents valid rollback snapshots (for example after `debug_undo`) from being discarded as stale.
- Gateway sends `game_action_enqueued` acknowledgements with `command_id`/`stream_id` for distributed actions.
- Gateway exposes `request_distributed_debug` -> `distributed_debug_info` to inspect:
  - current Redis runtime state version and recent command ids,
  - command stream entries,
  - pending consumer-group entries.
- Lobby membership resolution in `RoomManager` is presence-first (`PresenceService`), with legacy in-memory lobby map used only as compatibility fallback.

## 3) State and Storage

## 3.1 Persistent
- PostgreSQL (`backend/app/db_models.py`) for:
  - users/profiles/admin flags
  - collection/decks/progression/currencies
  - room and match records
  - replay artifact metadata

## 3.2 Runtime/Ephemeral
- Redis for:
  - distributed command streams (`game:stream:<game_id>`)
  - stream registry / consumer groups
  - dead-letter stream (`game:stream:dead_letter`)
  - pub/sub outbound channels
  - timer scheduling + due pop (`game:timers` ZSET + timer actor)
  - presence/session ephemeral keys

## 3.3 Replay data
- Replay metadata in DB.
- Replay payload supports local JSON artifact flow.
- Archive pipeline scaffolding exists (Redis queue + background archiver + pluggable object-storage provider).
- Object-storage-first archival is still not the default mode in every environment.

---

## 4) Security Model (Current)

- App authentication is Firebase-based.
- App admin authorization is managed in DB/profile flags (with bootstrap allowlist support).
- Redis authentication is now supported via `REDIS_PASSWORD` + authenticated `REDIS_URL`.
- RedisInsight currently uses:
  - Redis credentials (for DB connection),
  - host-level protection strategy (bind to localhost + SSH tunnel in production profile).

## 5) Deployment Model

## 5.1 CI/CD
- GitHub Actions builds/pushes backend/frontend images to GHCR.
- VM deploy updates compose bundle and restarts containers.

## 5.2 Production host
- Caddy reverse proxies:
  - `/api/*` and `/ws*` -> backend gateway
  - rest -> frontend
- Worker is internal (no public ports).
- Redis/Postgres are internal services.

## 6) Known Limitations (Important)

1. **Distributed runtime is transitional**  
   Worker currently reuses existing RoomManager/engine path through the local bridge; this is a migration stage, not final clean-slate worker core.

2. **Process-local runtime services still exist**  
   Legacy `fake_*`/runtime registry maps were removed from runtime paths, and per-game execution caches are now command-scoped/evicted, but authoritative gameplay still passes through process-local service objects (`RoomManager`, `LobbyService`, runtime singletons). Full multi-node determinism hardening is still in progress.

3. **Replay archive lifecycle is not final**  
   Full event-stream archival + object-storage-first retention pipeline is not yet the only mode.

4. **Distributed debug tooling is read-only**  
   The runtime debug panel/api exposes stream and pending diagnostics, but no admin command replay/requeue actions yet.

5. **DLQ operations are manual**  
   Dead-letter entries are persisted, but there is no admin UI/API yet for listing, replaying, or purging DLQ commands.

6. **Formal migration tooling**  
   Startup schema initialization is still app-driven. Full Alembic-style migrations are a pending improvement.

---

## 7) Practical Operational Guidance

- For local correctness testing of the distributed path, run the default compose stack (gateway + worker).
- For Redis inspection, use `redisinsight` profile and connect with the configured Redis credentials.
- For production RedisInsight access, prefer SSH tunnel to localhost-bound port instead of public exposure.

## 8) Priority Aspects To Address Next

1. **DLQ tooling and runbook**
   - Add admin/ops API to inspect dead-letter entries, requeue selected commands, and purge stale ones.
   - Add clear production runbook for error triage.

2. **Final OCC hardening**
   - Enforce expected-version semantics consistently across all mutating actions.
   - Add chaos tests for duplicate delivery + crash/restart + out-of-order attempts.

3. **Process-local service removal**
   - Remove remaining bridge dependence on in-process `RoomManager` state maps.
   - Move recovery/hydration sources fully to Redis + DB records.

4. **Replay archival completion**
   - Finish object-storage-first archive flow and retention policy.
   - Keep local file artifacts only as explicit dev mode.

5. **Observability**
   - Add metrics/tracing for command latency, claim/retry rates, DLQ rate, timer lag, and CAS conflicts.
   - Define SLOs and alert thresholds.

6. **Command-level operations tooling**
   - Add admin actions to requeue/purge specific pending/dead-letter messages.
   - Add command-by-command traceability from websocket ack (`command_id`) to stream entry and commit result.

7. **Multi-node production drills**
   - Validate multiple gateway + worker replicas with synthetic load.
   - Verify no cross-game leakage and predictable failover behavior.
